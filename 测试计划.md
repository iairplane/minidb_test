# 《测试计划》——Minidb（黑盒+白盒+大模型驱动）

## 1. 测试目标

1. 验证 Minidb 核心流程与功能点满足《需求说明文档》。
2. 同时完成黑盒测试与白盒测试设计、执行与分析。
3. 在自动化测试执行中达到并维持代码覆盖率 >=95%。
4. 形成可复用的大模型驱动测试工作流。

## 2. 测试范围

- 代码范围：`minidb/minidb.py`
- 重点模块：`Store`、`Model`、查询表达式系统、类型转换器、输出格式化函数。

## 3. 测试策略

### 3.1 黑盒测试策略

基于 FR-01~FR-14 进行需求驱动测试，覆盖以下方法：

1. **等价类划分**
   - 合法类：已注册模型、合法字段类型、唯一查询结果等。
   - 非法类：未注册模型、重复注册、非法关键字参数、非唯一 `get` 等。

2. **边界值分析**
   - 查询结果边界：0 行、1 行、多行。
   - `limit` 边界：不设、1、较大值。
   - 日期时间边界：含/不含微秒。

3. **决策表**（核心）
   - `save_or_update` 决策条件：
     - 条件A：`id is None`？
     - 条件B：对象已绑定 db？
     - 条件C：`smartupdate` 是否开启？
   - 覆盖各组合对应行为（插入/更新/无变化不更新/异常）。

### 3.2 白盒测试策略

1. **代码覆盖**
   - 使用 `pytest + pytest-cov` 对 `minidb.py` 统计。

2. **基本路径覆盖**
   - 重点路径：
     - `Store.__exit__` 正常提交路径 / 异常关闭路径。
     - `Store.query` 中 `select/where/group_by/order_by/limit` 多分支路径。
     - `Operation.argtosql/tosql` 多类型分派路径。
     - `Model.save/delete/get/load` 常规路径与异常路径。

3. **数据流分析（Def-Use）**
   - 关键变量：`sql,args`、`slots`、`pk`、`cache`。
   - 验证定义后使用在不同分支下保持正确（例如动态 SQL 拼装、缓存命中/失效）。

## 4. 自动化与大模型驱动工作流

1. 大模型输入：需求条目、未覆盖行、失败日志。
2. 大模型输出：
   - 测试点拆分；
   - 用例草案（黑盒/白盒）；
   - 覆盖率缺口补测建议；
   - 报告摘要。
3. 人工审核：
   - 对生成用例做可执行性校验；
   - 避免“只为覆盖而覆盖”的无效断言。
4. 自动执行：
   - `PYTHONPATH=. pytest -q`
   - `PYTHONPATH=. pytest -q --cov=minidb --cov-report=term-missing`

## 5. 通过准则

1. 全部测试用例执行通过。
2. 代码覆盖率 >=95%。
3. 核心流程相关测试全部通过。
4. 黑盒与白盒结果均可追踪到 FR 编号。

## 6. 风险与应对

1. 风险：第三方环境导入到 site-packages 版本导致覆盖统计偏差。
   - 应对：设置 `PYTHONPATH=.` 强制使用本地源码。
2. 风险：SQLite 事务中调用 `VACUUM` 报错。
   - 应对：测试中先 `commit()` 再 `vacuum()`。

## 7. 未来功能变更适配（2-3项）

1. **新增复合主键**
   - 黑盒：主键组合唯一性、冲突场景。
   - 白盒：`PRIMARY_KEY` 相关 SQL 拼接与分支路径重测。

2. **新增事务回滚策略**
   - 黑盒：异常中断后数据一致性。
   - 白盒：`__exit__` 异常路径、回滚调用路径。

3. **新增字段约束声明（唯一/非空）**
   - 黑盒：非法插入、更新冲突。
   - 白盒：建表/升级时约束 SQL 生成分支。
